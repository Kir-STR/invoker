# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Команды

- `npm test` — весь набор (`node --test "skills/invoke/test/*.test.mjs"`).
- Один файл тестов: `node --test skills/invoke/test/<name>.test.mjs`.
- Прогон CLI вручную: `node skills/invoke/helper.mjs resolve --modules <a,b>` (dry-run, печатает план) или `... apply --target <dir> --engines claude[,agents,gemini] --modules <a,b>`.

Сборки/линтера нет: чистый Node ESM (`.mjs`), без внешних зависимостей, тест-раннер — встроенный `node:test`. На Windows `node --test <каталог>/` ошибочно грузит каталог как модуль — поэтому скрипт использует quoted-glob.

## Архитектура

Плагин = **скиллы** (активная логика, `skills/`) + **статика** (`templates/`) + **control plane** (`modules.json` в корне).

**Поток `invoke` — гибрид NL + детерминизм.** `skills/invoke/SKILL.md` — тонкий оркестратор (NL): задаёт capability-вопросы, gated-подтверждения, отчёт. Вся механика — в Node-ядре `skills/invoke/lib/`, конвейер: `registry` (load + validate) → `resolve` (топосортировка `depends_on`, детект циклов/битых ссылок) → `blocks` (вырезание блоков) → `scaffold` (копирование/рендер/слепок). `helper.mjs` — тонкий CLI над ядром (`resolve`/`apply`).

**Главный инвариант: `invoke` копирует дословно, не синтезирует текст.** Правила копируются `copyFileSync` (байт-в-байт, не read+write-as-string — иначе BOM/кодировка поедут). Инструкц-файлы собираются *включением/исключением* размеченных блоков `<!-- module:<id> -->…<!-- /module -->`, а не генерацией. Любая правка, добавляющая «умную» генерацию текста в `invoke`, нарушает контракт.

**`modules.json` — единственный source of truth для `invoke`** (группы/дефолты/`depends_on`/`note`). Лежит в корне плагина и **в целевой проект не копируется** — туда пишется слепок `.invoker/modules.json` (`selected_modules` vs `resolved_modules` разведены намеренно).

**Служебная зона целевого проекта — `.invoker/`.** Вся invoker-специфичная инфра живёт здесь: слепок `modules.json`, инбокс `ideas_4_rules.md`, шаблон `retro-template.md`, черновики `retro/<branch>.md` (последние пишет будущий `retro`). `.claude/` остаётся под нативные Claude Code rules. Шаблон лежит вне подпапки `retro/`, чтобы не попасть под glob черновиков `.invoker/retro/*.md`.

**Multi-engine без транспайлера.** Три *ролевых* шаблона в `templates/instructions/` (CLAUDE = оператор, AGENTS = консультант, GEMINI = ревьюер), каждый содержит блоки всех модулей. `invoke` эмитит только выбранные движки. Авто-перевода одного в три нет.

**`SKILL.md` зовёт helper через `${CLAUDE_PLUGIN_ROOT}`** (переменная Claude Code, подставляется в содержимое скилла): при запуске cwd — целевой проект, а не каталог плагина, поэтому относительный путь до `helper.mjs` не разрешится.

**Что эмитит `apply`.** В целевой проект: дословные копии правил → `.claude/rules/<id>.md`; инструкц-файлы выбранных движков → в корень (`CLAUDE.md` всегда, `apply` форсит `claude` первым); служебка → `.invoker/` (`ideas_4_rules.md` всегда; `retro-template.md` — только если выбран `retro-loop`) + слепок `.invoker/modules.json`. Валидация движков идёт до любых записей — частично инициализированный проект исключён.

## Синхронизация при добавлении/изменении модуля

Целостность охраняет инвариант-тест `skills/invoke/test/consistency.test.mjs`: множество блоков `<!-- module:<id> -->` в **каждом** из 3 шаблонов == множество id в `modules.json` == файлы `files` существуют под `templates/`. Добавляя модуль, синхронно правь: `modules.json` + `templates/rules/<id>.md` + блок во **всех трёх** инструкц-шаблонах — иначе тест падает.

**Тест не охраняет** ещё две вещи — их правь руками: (1) чеклист групп в `skills/invoke/SKILL.md` (перечисление модулей по группам, отметка recommended); (2) **версию в `.claude-plugin/plugin.json`** — поднимать на каждый функциональный релиз. Авто-обновление плагина в Claude Code резолвит по строке `version`, не по коммиту: без bump'а маркетплейс с `autoUpdate: true` подтянется, а установленный плагин останется на старом коммите.

## Конвенции

- `modules.json` — строгий JSON (без комментариев/висячих запятых): его читает `JSON.parse`.
- Проза — русская; идентификаторы, имена файлов и плейсхолдеры `{{...}}` — ASCII.
- `.gitattributes` форсит `eol=lf` на `*.md`/`*.tmpl` — критично: блок-фильтрация построчная, посторонний `\r` дал бы смешанные переводы строк в рендере.
- Контент `templates/rules/` дистиллирован из eve и универсализован: проектная специфика → нейтральные формулировки. `safety.md` — намеренно скелет (заголовки + TODO), без переноса доменных инвариантов. Правя правила, сохраняй дисциплинарный текст дословно и не вноси eve/доменную специфику.
- Плейсхолдеры `{{...}}` оставляем **только** там, где значение проектно-уникально и нет разумного дефолта. Описание проекта (имя/стек/конвенции) — не дело invoke, его в шаблонах нет. Дефолты вместо плейсхолдеров: `globs` → `**/*`, ветка → `main`, PR-инструмент → `gh`, бюджеты PR → числа (400/800/15), имя ревью-бота → обобщённое `review-bot` (с подсказкой заменить). После этого проектно-уникальных `{{...}}` в правилах не остаётся; каркасы `safety`/`architectural-invariants` заполняются под проект вручную, но это TODO-разделы, не плейсхолдеры. invoker — универсальный набор: про промпты/LLM-артефакты в правилах не говорим. `{{BRANCH}}` в `.invoker/retro-template.md` — runtime-плейсхолдер будущего `retro`, не для оператора.
- `AGENTS.md` (в корне) задаёт роль Codex как предпубликационного адвизора для этого репо — не дублируй его в CLAUDE.md.
