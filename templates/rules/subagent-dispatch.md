# Subagent dispatch — дисциплина

Применять при диспатче implementer-субагентов через ваш subagent-скилл (или эквивалент).

## Implementer обязан STOP при divergence от плана

Если по ходу исполнения plan-инструкция не работает (команда падает с ошибкой, не предсказанной планом; менеджер пакетов подтягивает другие мажоры; конфиг проходит локально, но не в CI) — implementer обязан остановиться («заблокирован» / «нужен контекст»), **не** пытаться чинить автономно. Корректирующая работа применяется только после согласования с оператором.

**Статус «сделано с оговорками» со строкой в commit-body не заменяет «заблокирован»** при техническом расхождении. Даже если фикс «очевидный» (1–2 строки) — STOP и эскалация. Цена несогласованной «маленькой» автономии = ретроспективный sync-коммит постфактум + потерянная возможность синхронизации plan/spec до коммита.

**Автономный tooling-fix — то же нарушение, даже тривиальный.** Дрейф расширения, причуды линтера, соответствие формату — авто-фикс силами implementer'а даже на уровне 1 символа = нарушение процесса. Озвучить постфактум ≠ одобрение. Восстановление — ретроспективный `docs: spec sync` в plan § Deviations log (категория tooling), не молчаливое принятие в feat-коммите. Заданный оператором числовой порог (LOC ≥ N, findings ≥ N) при срабатывании — автоматический «заблокирован», не «сделано с оговорками»; порог — количественный гейт, не субъективная оценка.

## Package version drift

Когда менеджер пакетов подтягивает мажоры, отличные от plan/spec — implementer **обязан** остановиться. Оператор решает: принять последний стабильный или откат к плановым мажорам.

## Implementer'ы — строго последовательно

Implementer-субагенты диспатчатся **строго последовательно**, с ревью между задачами: параллельный запуск нескольких implementer'ов запрещён — конфликты. Параллельный батч задач — нарушение процесса, даже если дизъюнктность файлов случайно спасает от конфликта. SHA-границы в брифе ревьюера фиксировать **после** коммита implementer'а (BASE = родитель, HEAD = фактический SHA), не предсказывать — worktree двигается под параллельно запущенными ревьюерами, предсказанные SHA-диапазоны не резолвятся. Параллелить допустимо только **read-only** агентов (simplify / spec / quality) на **уже закоммиченном** коде.

## Изменения shared-контракта — co-commit потребителя + полная проверка

Шаг плана, делающий поле **обязательным** / расширяющий shared-контракт (тип, схему, интерфейс) с существующими потребителями, **обязан** co-commit'ить потребителя + producer-wiring в той же задаче/коммите — иначе промежуточный коммит красный по проверке. Отделять wiring в позднюю задачу допустимо только если поле опциональное на переходный период. Верификация — полная статическая проверка / сборка (`tsc --noEmit`, `mypy`, компиляция — или эквивалент), **не** только unit-тесты (тестовые рантаймы не проверяют межмодульную совместимость типов/контрактов).

## Mechanical verification > display / memory reading

Когда инструкция оператора либо собственная гипотеза из памяти противоречит механическому свидетельству (git output, содержимое файла, ответ API) — агент обязан возразить с фактическим доказательством, не молча следовать инструкции. Возражение работает в обе стороны: оператор может опровергнуть утверждение агента; агент обязан опровергнуть инструкцию оператора, если её опровергают факты. Память — стартовая подсказка, не авторитетная хронология: всегда сверять с первоисточником (git / файловая система / ответ API) до триажа / коммита / публикации.

Дисциплина: проверка первоисточника — **до** триажа / коммита / публикации. Цена — 1 query; предотвращает каскад ошибок и синхронизацию задним числом.

## Umbrella sensitive-paths pre-flight

Для зонтичного плана (≥ 2 child-PR) — sensitive-paths sweep по каждому glob из `CLAUDE.md § «Sensitive paths gate»` ДО старта исполнения; > 0 совпадений → true-positive (план `review-approved` для child-PR) либо false-positive (narrow-CI fix отдельным PR заранее, ДО исполнения).
