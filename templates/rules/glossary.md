# Глоссарий — конвенции терминологии

Языковая дисциплина для **содержания** правил, документации, UI, описаний issue/PR. **Имена** (slug'и веток, имена файлов, импорты, ключи YAML/JSON, идентификаторы кода) — отдельная политика, всегда ASCII.

Цель — снять word-level code-switching в основном языке документации, сохранив компактность там, где английский плотнее по смыслу.

## Whitelist — English-OK внутри основного текста

1. **Идентификаторы** — хеши коммитов, пути файлов, имена команд и флагов, имена функций/переменных/классов, ID issue/PR, имена workflow / job / step CI.
2. **Proper nouns инструментов** — названия конкретных инструментов, платформ, библиотек (GitHub, Docker, TypeScript и т.п.).
3. **CLI-команды в обратных кавычках** — `git rebase`, `npm run lint` и т.п. (как команды/действия, не как глаголы).

## Применение

- В документации берём основной язык из таблицы переводов ниже. Если перевод неоднозначен на месте — допустимо в скобках продублировать английский.
- Тривиальные когнаты (`session`, `host` и т.п.) в таблицу не вносим — их покрывает дефолт «берём основной вариант»; таблица несёт только неочевидные переводы и keep-English решения.
- Whitelist английских классов и keep-English список не переводим даже при наличии эквивалента.
- Имена (slug'и веток, ASCII-файлы, импорты, ключи) — латиница.
- Имена файлов-контрактов для движков/внешних ревьюеров (если такие используются) остаются английскими (carve-out).
