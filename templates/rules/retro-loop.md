# Retro loop — цикл эволюции правил

Идентификационная фича invoker: правила не статичны, они эволюционируют через retro-цикл.

## Контракт цикла

```
observation → .invoker/retro/<branch>.md → .invoker/ideas_4_rules.md (n-count) → promote → cleanup
```

Вся служебная инфра цикла живёт в `.invoker/` (рядом со слепком `modules.json`); `.claude/` остаётся под нативные правила.

1. **Observation.** По ходу работы в worktree агент фиксирует наблюдения: что сработало, что не сработало, что стоит закрепить правилом.
2. **`.invoker/retro/<branch>.md`.** После merge сессии — обязательный последний шаг: записать предложения правок `CLAUDE.md` / `.claude/rules/*.md` в `<main-checkout>/.invoker/retro/<branch>.md` (неотслеживаемый файл; шаблон — `.invoker/retro-template.md`). Дублировать выводом на экран.
3. **`.invoker/ideas_4_rules.md`.** Промежуточная зона. Записи из retro-черновиков накапливаются здесь через `ivk retro` (или эквивалентный скилл). Файл локальный, gitignored — рабочее состояние мастер-сессии.
4. **n-count → promote.** Когда идея набирает достаточную поддержку (повторяется из разных retro) — оператор принимает решение о промоции: идея превращается в точечную правку правила через PR.
5. **Cleanup.** После промоции черновик `.invoker/retro/<branch>.md` и запись в `.invoker/ideas_4_rules.md` удаляются.

## Дисциплина retro-файла

- Формат: свободный, но структурированный (секции по типу — «что закрепить», «что убрать», «что уточнить»).
- Не коммитить retro-файлы в репо (gitignored).
- Не пропускать шаг: ретроспектива без retro-файла — потеря сигнала.
- Ретроспектива описывает **конкретные прецеденты**, не общие ощущения: что произошло, какое правило не хватало или оказалось излишним.

## Дисциплина promote

- Промоция — через PR, не прямым коммитом в `main` (если правило затрагивает чувствительные пути).
- Правка правила сопровождается удалением или пометкой «done» в `.invoker/ideas_4_rules.md`.
- Оператор явно одобряет промоцию (гейт).
