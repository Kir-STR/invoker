# Operator-gate дисциплина

Явное подтверждение оператора авторизует серию **обратимых** шагов одной операции — один гейт на серию; обратимые шаги батчить. Не предавторизовывать гейт, необходимость которого ещё не подтверждена наблюдаемым состоянием.

Необратимое и outward-facing — отдельный гейт каждому:

- **outward-facing** (создание публичного PR, публикация, деплой) — отдельный гейт, буквальное «да»; «готовность подтверждена» (координатор / ревьюер) ≠ авторизация; create ≠ merge;
- **обычный merge** — отдельный явный гейт (буквальное «мержь»);
- **деструктивные / высокорисковые** (admin override merge, `push --force`, `reset --hard`, удаление веток/данных) — отдельный гейт каждый раз, форма **report+STOP**, **не** UX-модаль: осознанный выбор оператора, не механический пересказ;
- **недеструктивные гейты** (публикация комментария, установка лейбла) — UX-запрос допустим.

**Эшелонированная защита:** обход автоматического гейта (admin override, label-override) не каскадируется на соседние инварианты — явная перепроверка соседних гейтов в audit-trail.

**Gate 1 / Gate 2 split.** Когда действие состоит из публикации (Gate 1, недеструктивный класс) и последующего исполнения (Gate 2, деструктивный класс) — каждый гейт требует **независимого** буквального «да». Gate 2 не наследуется из Gate 1. Между Gate 1 и Gate 2 — обязательный свежий пакет перепроверок: обновить состояние из источника правды, проверить статусы CI, пересчитать метрики.
