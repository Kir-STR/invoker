# PR policy — политика веток и PR

Цель — каждый PR остаётся reviewable для автоматического ревьюера и удобным для ручной проверки.

## Основные правила

- `#N` в тексте PR/коммитов — только проверенные ссылки. Никогда не использовать `#N` для эфемерных / backlog-ID в title/body PR и subject коммитов — платформа может рендерить `#N` backreference на чужие закрытые PR/issue.
- Один PR = одна цель.
- Ветки от `main`, merge только через PR.
- Merge через `gh` с опцией `--rebase` (если не настроено иначе). Не использовать `--squash` или merge-commit без явного согласования.
- Merge разрешён только при: все CI-проверки зелёные; оператор явно сказал «мержь».

## Simplify pass перед PR

Любой план (ваш скилл планирования или эквивалент) обязан содержать шаг «simplify pass» непосредственно перед шагом push + создание PR: запуск шага упрощения (`/simplify` или эквивалент), согласование findings с оператором, отдельный коммит на применённые правки. Когда триаж simplify-pass возвращает 0 применённых правок — коммита нет. Пустой коммит `refactor: simplify pass` ради формального учёта засоряет историю ветки; дисциплина удовлетворена самим фактом выполненного триажа.

## Sync spec/plan при отклонениях от плана

Если по ходу исполнения возникли решения, расходящиеся со spec'ом или планом — обязательный коммит `docs: spec sync` перед simplify pass, приводящий spec и (при необходимости) смежные документы в соответствие с фактом. Plan дополняется секцией `## Deviations log` с явным перечислением отклонений и причин.

Классы docs-коммитов: `docs: spec sync` — расхождение факта с записанным spec/plan; `docs: correct <X>` — точечная правка до исполнения; `docs: record <X>` — фиксация состояния; `docs: clarify <X>` — точность формулировок; `docs: fill <section>` — заполнение шаблонных секций. Смешение классов размывает audit-trail.

## Бюджет PR

**По умолчанию:**
- target: до `400` reviewable lines
- hard cap: `800` reviewable lines **и** `15` reviewable files
- один PR = одна цель

**Reviewable diff** считается без lock-файлов, сгенерированных файлов и артефактов сборки.

**Метрика:** added + deleted через `git diff --numstat`, не сырые строки unified diff.

## Sensitive paths gate

Зоны, в которых PR требует отдельного согласования с оператором. Список чувствительных путей — в `CLAUDE.md § «Sensitive paths gate»` проекта. При работе в этих зонах держать PR ещё меньше обычного бюджета.

**Предварительная проверка перед любой «мелкой/doc-only» правкой:** проверить, что путь файла не пересекается с чувствительными зонами. Если пересекается — обязательно ветка + worktree + PR, даже если содержание правки — две строки конфига. Direct-to-main для чувствительного пути исключений не имеет.

## Review loop

**Базовое правило:** ничего не править автономно по review-комментам `review-bot` (обобщённое имя автоматического ревьюера — замени на свой: Gemini Code Assist, Copilot, Claude review и т.п.). Любая правка после review требует явного подтверждения оператора. Никаких авто-итераций push → review → fix → push. После разбора findings — показывать оператору каждый finding с предлагаемой реакцией:
- **Срочно править сейчас**
- **В backlog** — записать finding в backlog под target-секцию
- **Отвергнуть** — с обоснованием через комментарий к PR
